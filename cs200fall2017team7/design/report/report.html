System Description

Chocoholics Anonymous (ChocAn) is an organization dedicated to helping people addicted to chocolate. Our organization has been awarded the contract to write the ChocAn data processing software. This software will need to be able to validate that a person with a ChocAn card is indeed a current member in good standing of ChocAn by having a provider swipe the member card on a provider terminal. Additionally, the system provides support for billing, updates members, and automatically sends customized reports to members, providers, and managers on ChocAn information relevant to them.

Glossary

  • Electronic Funds Transfer (EFT) data: data that is contained in the weekly report that providers receive when they've billed ChocAn.
  • Service Code: a code that corresponds to a specific service provided by ChocAn.
  • Monthly fee: a monthly subscription fee paid by members to ChocAn.
  • Member: those who pay a monthly fee to ChocAn to receive health care in return.
  • Provider: a health care professional who provides services to ChocAn members.
  • Terminal: a computer similar to a credit card device in a shop.
  • Comments: if necessary, a provider can write comments in their records to describe a service they provided.
  • Provider Directory: providers use this directory to look up the appropriate six-digit service code corresponding to the service they provided. Providers can request this directory as an email attachment via the provider terminal.
  • Summary Report: contains information needed to update financial records.
  • Chocoholics Anonymous (ChocAn): an organization dedicated to helping people addicted to chocolate in all its glorious forms.
  • Acme Accounting Services: a third-party organization responsible for financial procedures. The Acme computer acts on our data center each evening.

System Operations Paragraph

The system will generate reports based on data stored in the system and time and will send the reports via email. EFT’s will be stored onto a disk, and other data will be stored in our system. We will need a server for the records and the provider’s terminals. Our system will interact with the provider’s terminal system for member validation and billing, with Acme’s system when processing ChocAn membership fees, and with banking computers to credit providers’ bank accounts. Providers validate and bill members by swiping their cards on a provider terminal. For all other use cases, the system may be interacted with remotely or through local computers.

Discussion of Design

The design for our software system has high cohesion in a few ways. Each class was made to support one main operation. We used a high degree of generalization to separate these specific classes by each user that would be affected by or need that class's functionality. The classes would work together to perform their separate functions for their user. Our design strived for little coupling between classes. With each class designed for a specific function, each class needs only a limited amount of interaction with other classes and their data. The classes will use each other for a singular purpose, and a class won’t use the other functions or methods of another class until absolutely necessary. This makes for a well-designed system since classes can be easily altered using inheritance, allowing support for a new class.
